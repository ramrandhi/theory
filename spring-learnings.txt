1). SPRING IOC (INVERSION OF CONTROL) CONTAINER :

The Spring IoC container is responsible for managing the beans and their dependencies in a Spring Boot application. The IoC container includes:

Sping boot container types :

a). Bean Factory - 
	1. bean factory is the simplest container in spring where it provides the basic functionality like dependency injection and managing beans.
	2. Bean factory lazy loads beans (only when they are requested and resolves the dependencies.
	3. bean Factory is primarily used for small scale applications where all the features of application context is not required.
	4. Example: XmlBeanFactory (deprecated) and programmatic configuration via BeanFactory.

b). Application Context - (Application context is the extension of bean factory that adds more enterprise level features.
	1. Application context eagerly loads all the beans at startup, unlike BeanFactory.
	2. Supports event propagation, declarative mechanisms for creating beans, internationalization (I18N), and AOP (Aspect-Oriented Programming).
	3. Offers additional functionality such as bean post-processing, lifecycle management, and integration with Spring’s web and transaction services.
	
	*. Types of application context - 
	1. AnnotationConfigApplicationContext - loads the context using annotations (Typically used in spring boot).
	2. ClassPathXMLApplicationContext - loads the context from XML Configuration file.
	3. WebApplicationContext - A specialized ApplicationContext for web applications, integrated with Spring's MVC framework.

	Mappings -> 4 types (i). one to one, (ii). one to many, (iii). many to one, (iv). many to many.
	
	@OneToOne -> This is a one-to-one relationship between two entities. In the database, one row in the first table corresponds to exactly one row in the second  table. The foreign key is typically stored in one of the tables (the parent table).
	
		Example : -> Here we referenced Student and Laptop where student having laptop object with onetoone relation to referene laptop class.
		
			@Entity
			public class Student {
				
				@Id
				@GeneratedValue(strategy = GenerationType.IDENTITY)
				@Column(name="ID")
				private Integer id;
				
				private String name;
				private Integer marks;
				
				@OneToOne(cascade = CascadeType.ALL, orphanRemoval = true)
				@JoinColumn(name = "LAPTOP_ID", referencedColumnName = "ID")
				private Laptop laptop;
			}
	
	
	@OneToMany -> In this relationship, one entity (parent) can be related to multiple instances of another entity (child). The foreign key is stored in the child table to reference the parent.
	
		Example : -> One Student can have multiple Laptops, and the foreign key STUDENT_ID is stored in the Laptop table to establish the relationship.
		
			@Entity
			public class Student {
				
				@Id
				@GeneratedValue(strategy = GenerationType.IDENTITY)
				@Column(name="ID")
				private Integer id;
				
				private String name;
				private Integer marks;
				
				@OneToMany(cascade = CascadeType.ALL, orphanRemoval = true)
				@JoinColumn(name = "STUDENT_ID")
				private List<Laptop> laptop = new ArrayList<>();
			}
	
	@ManyToOne  -> This is the reverse of the One-to-Many relationship. Many child entities are related to one parent entity. The foreign key is stored in the child table.
	
		Example : -> Many Laptop entities can be associated with one Student, and the foreign key STUDENT_ID is stored in the Laptop table.
		
			@Entity
			public class Student {

				@Id
				@GeneratedValue(strategy = GenerationType.IDENTITY)
				@Column(name="ID")
				private Integer id;
				
				private String name;
				private Integer marks;

				@OneToMany(mappedBy = "student", cascade = CascadeType.ALL, orphanRemoval = true)
				private List<Laptop> laptops = new ArrayList<>();
			}
		
			@Entity
			public class Laptop {

				@Id
				@GeneratedValue(strategy = GenerationType.IDENTITY)
				@Column(name="ID")
				private Integer id;
				
				private String model;
				
				@ManyToOne
				@JoinColumn(name = "STUDENT_ID")  // Foreign key in the "Laptop" table refers to "Student" ID
				private Student student;
			}

	HIBERNATE CACHING: -> Hibernate provides several caching mechanisms to optimize the database by reducing the number of database queries. there are two levels of caching in hibernate. (i). First-Level cache (session cache), (ii). Second-Level cache (SessionFactory cache), (iii). quere cache.

	1. First-Level cache (session cache) -> This is a cache technique which is enabled by default and cannot disable this cache. In this caching technique when inside an api call which we can take it as session when opened to make transactions when the api is fetching the data from same entity the data will be cached inside the session and that cache provides data without making db call until the session is closed. In between the session if the same entity is called multiple times hibernate will not query the database and returns data from the stored cache from that session.
		(a). Transaction-scoped.
		(b). Automatically managed by Hibernate.
		(c). Cannot be disabled.
		
	2. Second-Level Cache (SessionFactory cache)  -> Second level cache technique is technique in which this cache is available for different sessions in the application until or unless the data is updated and we need to set that using @cache(usage = CacheConcurrencyStrategy.READ_WRITE) with this when the cache data is overwritten in the database then cache is marked as needed to be refreshed and this cache is not available until cached again or queried to database for updated data.
	-> This is a Entity level cache so only entities marked with @Cacheable and @Cache(usage=CacheConcurrencyStrategy.READ_WRITE) can be cached to SessionFactory.
	
		(a). SessionFactory-scoped.
		(b). Can be shared across sessions.
		(c). Helps to avoid repeated queries across multiple transactions.
		(d). Must be explicitly configured.
		
		1. READ_ONLY:
			Best for data that doesn't change (e.g., reference or lookup data).
			Suitable for highly read-intensive scenarios.
			Updates to cached entities are not allowed.
			
		2. READ_WRITE:
			Allows reading and writing of cached entities.
			Ensures strong consistency by using transactional guarantees, typically with locks.
			
		3. NONSTRICT_READ_WRITE:
			Data can be read from the cache and updated, but it doesn't guarantee strong consistency.
			Some stale data may be present if one session updates the entity and another reads before the changes are flushed.
			
		4. TRANSACTIONAL:
			Provides the strongest consistency guarantees using underlying transaction management.
			Usually, it integrates with JTA (Java Transaction API) to maintain cache consistency in distributed systems.
			
		-> We need to use third-party cache provides for second_level cache most used are -> 1. EHCache, 2. Redis, 3. Infinispan, 4. Hazelcast

			Use Case:

			Best suited for optional dependencies (dependencies that might not always be required).
			Allows you to modify dependencies post-object creation.
			Useful when dependencies can change or are dynamically configured at runtime.
			Advantages:

			More flexible; you can set or modify dependencies as needed.
			Better for complex configurations where some dependencies may be optional.
	-> Constructor Injection: If there are values that are not being initialized dont go for constructor injection.
	
		Definition: The dependency is provided at the time of object creation through the constructor.

			Use Case:

			Best for mandatory dependencies (those required to create a fully initialized object).
			Should not be used when certain dependencies might remain uninitialized or optional.
			Advantages:

			Ensures immutability: once the object is constructed, its state cannot be changed.
			Guarantees that the object is in a fully initialized state after creation.
			Drawback:

			If there are optional dependencies, constructor injection is not suitable because it forces you to provide all dependencies upfront.
	
	@Autowired : 
			@Autowired is a Spring annotation used for dependency injection. It allows Spring to resolve and inject the necessary beans automatically into a class.
			
			1. Injects Dependencies Automatically:

				Spring scans the context and finds a matching bean by type (default behavior).
				It then injects the bean into the annotated field, constructor, or setter method.
			2. Injection Types:

				Field Injection: Injecting directly into class fields.
				Setter Injection: Injecting through a setter method.
				Constructor Injection: Injecting through a constructor.

					}
				}

		Key Points
		
			1. Default Behavior: By default, @Autowired performs injection by type.

			->	If multiple beans of the same type exist, you can use the @Qualifier annotation to specify the bean.
			
			Example : 
				@Autowired
				@Qualifier("specificBean")
				private MyRepository myRepository;
				
				
			2. Required vs. Optional:
			
				By default, @Autowired is required, meaning Spring will throw an exception if no bean is found.
				To make it optional, set required = false
			Example : 
				@Autowired(required = false)
				private MyRepository myRepository;
				
			3. Circular Dependencies: If two beans depend on each other, Spring may throw a circular dependency error. Consider refactoring your design or use @Lazy.
			
			4. Primary Bean: If multiple beans of the same type exist and no @Qualifier is provided, you can mark one bean as the primary one using @Primary:
			
				Example : 
				@Primary
				@Component
				public class DefaultRepository implements MyRepository {
					// Implementation
				}





	
	@Primary :
		In Spring Framework, the primary annotation refers to @Primary, which is used to mark a bean as the preferred candidate for dependency injection when multiple beans of the same type exist.

		Key Features of @Primary:
		
		1.Default Bean Selection:
			When multiple beans of the same type are defined in the Spring context, @Primary helps Spring decide which bean to inject by default without requiring further qualification.
		
		2. Override with @Qualifier:
			Even if a bean is marked as @Primary, you can override the default selection using the @Qualifier annotation.
			
		3. Usage in Configuration Classes:
			Commonly used in @Configuration classes to specify which bean should be the primary choice when there are multiple beans of the same type.
			
		Exmaple : 
		
			@Configuration
			public class AppConfig {

				@Bean
				public DataSource dataSource1() {
					return new HikariDataSource(); // Bean 1
				}

				@Bean
				@Primary
				public DataSource dataSource2() {
					return new BasicDataSource(); // Bean 2, marked as Primary
				}
			}
			
			Injecting beans with above configuration
			
			@Service
			public class MyService {

				private final DataSource dataSource;

				public MyService(DataSource dataSource) {
					this.dataSource = dataSource; // Injects dataSource2 (marked as @Primary)
				}
			}
			
		Overriding @Primary with @Qualifier:
		
			@Service
			public class AnotherService {

				private final DataSource dataSource;

				public AnotherService(@Qualifier("dataSource1") DataSource dataSource) {
					this.dataSource = dataSource; // Explicitly injects dataSource1
				}
			}
			
		When to Use @Primary:
			Use @Primary when there is a logical default bean, and you want to avoid specifying @Qualifier everywhere.
			Avoid overusing it in scenarios where it could lead to confusion or ambiguity.

EXCEPTION HANDLING IN SPRING BOOT : Spring internally has SpringErrorHandler class that can handle the minor exceptions but for specific exception that need to be implemented we need to use @ControllerAdvice to framework level for spring to intercept all the exceptions thrown at controller level.

-> @ControllerAdvice -> Handles Exceptions from both RestController and Controller
-> @ControllerAdvice(annotations = RestController.class) -> Handles Exceptions only comes from RestControllers.
-> @ControllerAdvice(annotations = Controller.class) -> Handles Exceptions only from Controllers
-> @Order(Ordered.HIGHEST_PRECEDENCE) -> this gives the higest priority to this current class annotated with @ControllerAdvice to handle i mean high priority.
-> @ExceptionHandler(Exception.class) -> this is a method level annotation used to specify the type of exception that method should be handling

1. It is a specialized annotation used to handle exceptions thrown by controllers.
2. It works globally unless scoped to specific packages or controllers.
3. You can define multiple exception handlers for different exception types.

@Order(Ordered.HIGHEST_PRECEDENCE)
@ControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(IllegalArgumentException.class)
    public ResponseEntity<String> handleIllegalArgumentException(IllegalArgumentException ex) {
        return ResponseEntity.status(HttpStatus.BAD_REQUEST).body(ex.getMessage());
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<String> handleAllExceptions(Exception ex) {
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR).body("An unexpected error occurred");
    }
}

ZIPKIN : 
	-> Zipkin is a distributed tracing system that helps gather and visualize latency data across microservices. Below is an explanation of how it works internally, along with a diagram.
	
	Zipkin Internal Components
1. Instrumented Client / Service
	Applications (microservices) are instrumented with Zipkin libraries.
	These libraries generate tracing data and send it to Zipkin.
	Communication can be via HTTP, Kafka, RabbitMQ, or gRPC.
	
2.Zipkin Collector

	Receives trace data from instrumented services.
	Data can be received via HTTP, Kafka, or other transport protocols.
	Stores trace data in a database.
	
3.Storage Backend
	Zipkin supports multiple databases like MySQL, Cassandra, Elasticsearch.
	Stores the trace information for later retrieval.
	
4. Zipkin Query Service
	Exposes APIs for fetching trace data.
	Queries stored trace data from the backend.
	
5. Zipkin UI
	A web-based interface to visualize traces.
	Helps developers analyze request flow and identify bottlenecks.
	
=> Zipkin Internal Flow
	A request is initiated from one service to another.
	The Zipkin instrumentation records metadata (trace ID, span ID, timestamps).
	Tracing data is sent to Zipkin Collector.
	Zipkin Collector stores the traces in the storage backend.
	Zipkin Query Service fetches the trace data when requested.
	Zipkin UI visualizes the complete request flow.

CIRCUIT BREAKERS : 

	A circuit breaker is a design pattern commonly used in distributed systems to enhance the resilience and fault tolerance of applications. It acts as a safeguard to prevent an application from repeatedly trying to perform an operation that is likely to fail, thus avoiding unnecessary strain on system resources.

	How it Works:
	The circuit breaker monitors the interactions between services and tracks failures. It has three main states:
	
	Closed:
	In this state, the circuit breaker allows requests to flow through to the service.
	It monitors the success and failure rates.
	If failures exceed a predefined threshold within a specified time, the circuit breaker trips to the Open state.
	
	Open:
	In this state, the circuit breaker blocks all requests to the service.
	This prevents overloading the failing service and allows it time to recover.
	After a certain time, the circuit breaker transitions to the Half-Open state to test the service.
	
	Half-Open:
	In this state, the circuit breaker allows a limited number of test requests to the service.
	If these requests succeed, the circuit breaker transitions back to Closed.
	If they fail, it goes back to the Open state.
	Benefits of Circuit Breakers:
	Improved Resilience: Prevents cascading failures in distributed systems.
	Resource Optimization: Reduces load on failing services and frees resources for other tasks.
	Faster Recovery: Allows failing services time to recover before resuming normal operation.
	Better User Experience: Provides fallback responses rather than letting requests hang indefinitely.
	Real-World Analogy:
	A circuit breaker in an electrical circuit cuts off power to prevent overload and damage. Similarly, a software circuit breaker "trips" to protect a system from continuous failed interactions.

	Implementation in Spring Cloud:
	Spring Cloud provides built-in support for circuit breakers via libraries like:

	Resilience4j: A lightweight and modular framework for circuit breakers and other fault tolerance patterns.
