JDK - java development kit.
    - contains jre, jvm and other class loader files for building applications.

JRE - java run time environment
    - it contains jdk, class libraries and runtime utilities.


JVM - java virtual machine
    - loads .class files (bitecode)
    - features like garbage collection
    - ensures platform independence.
    - converts byte code to machine understandable code using jit compiler

Main Method in Java
public static void main(String[] args) {}

public - to make it accessable from anywhere
static - reference through class not through object as jvm will be having a overhead if object creation is required.
void - main method doesnot return any return type
main - this is the entry point of the method
String[] args - main method takes a array of arguments from command line

GARBAGE COLLECTION:
 - it mainly consists of 3 parts. 
 - 1. young generation, 2. old generation, 3. meta data
 - 1. YOUNG GENERATION: All the new objects will be created here.
 - 2. OLD GENERATIN: All the objects that are in contact with threads for longer time will be moved to old generation.
    - here accessing objects is costly because it is heavy and has lot of old data present inside it.
 - 3. metaspace - introduced in java 8 (before it was permgeneration) it used to store class data.
TYPES OF GARBAGE COLLECTORS:
 - 1. Serial Gc - (-XX:+UseSerialGC) - Uses single thread for GC, Suitable for small applications, Causes long pause times, Mostly used in single-core or low-memory environments
 - 2. Parallel GC - (-XX:+UseParallelGC) - Uses multiple threads for GC, Focuses on high throughput, Causes stop-the-world pauses, Default GC in Java 8.
 - 3. Concurrent mark sweep (CMS) GC (Deprecated) - (-XX:+UseConcMarkSweepGC) - Minimizes pause time, Runs most GC phases concurrently, Causes fragmentation, Removed in Java 14
 - 4. G1 GC (default gc) - (-XX:+UseG1GC) - Heap divided into regions, Predictable pause times, Concurrent + parallel phases, Default GC from Java 9 onwards
 - 5. Z GC - (-XX:+UseZGC) - Ultra-low latency GC, Pause times typically < 10 ms, Uses colored pointers & load barriers, Requires large memory, Scales to multi-TB heaps
 - 6. Shenandoah GC - (-XX:+UseShenandoahGC) - Low-latency GC similar to ZGC, Concurrent compaction, Slightly higher CPU usage, Supported mainly in OpenJDK

 -> In the context of Serial Garbage Collector, “long pause times” means the application completely stops executing business logic for a noticeable duration while garbage 
 collection runs. This is also called a Stop-The-World (STW) pause.

| GC Type    | Pause Time | Throughput | Heap Size  | Default        |
| ---------- | ---------- | ---------- | ---------- | -------------- |
| Serial     | High       | Low        | Small      | Old client JVM |
| Parallel   | High       | Very High  | Medium     | Java 8         |
| CMS        | Low        | Medium     | Large      | Deprecated     |
| G1         | Medium     | High       | Large      | Java 9+        |
| ZGC        | Very Low   | High       | Very Large | No             |
| Shenandoah | Very Low   | Medium     | Large      | No             |

example usage in spring application : 
java \
  -Xms2g \
  -Xmx2g \
  -XX:+UseG1GC \
  -XX:MaxGCPauseMillis=200 \

Exception Handling
 - Exceptions are of two types.  1. exceptions and 2. erros
 - 1. Exceptions
    - 1. run time exceptions - nullpointer exceptions, index out of bound exceptions etc.
    - 2. compile time exceptions - SQLException exceptions, file not found exceptions etc.
 - 2. Errors
    - these occur when something is wrong with the hardware level or jvm level.


Throwable
├── Error
│   ├── OutOfMemoryError
│   ├── StackOverflowError
│   └── VirtualMachineError
│
└── Exception
    ├── Checked Exceptions
    │   ├── IOException
    │   │   └── FileNotFoundException

    │
    └── Unchecked Exceptions (RuntimeException)
        ├── NullPointerException
        ├── IllegalArgumentException
        ├── IndexOutOfBoundsException
        └── ArithmeticException

1. Throwable is the top-level superclass.
2. Error: Serious problems that applications should not try to catch.
3. Exception: Conditions that applications might want to catch.
   1. Checked Exceptions: Must be declared or handled.
